# 睡眠恢复评估项目 MVP 分计划

## Summary

本计划基于现有文档 [sleep-recovery-project-plan.md](file:///c:/Users/AppleXian/Desktop/Swift%20Playground/docs/sleep-recovery-project-plan.md) 制定，只覆盖当前优先级最高的 `MVP` 阶段，并按里程碑作为交付节点组织实施。

本次计划将原始文档中的 `MVP-1` 到 `MVP-4` 细化为可执行的分计划，并吸收本轮补充决策：

1. `Kaggle` 公开睡眠数据作为 MVP 主训练数据来源
2. `Swift` 端只接收预计算特征包和展示用时序，不在端侧重复做特征提取
3. `XGBoost / LightGBM` 作为 Core ML 主部署目标
4. `Logistic Regression`、`Random Forest` 作为对比模型，用于体现问题分析、建模取舍和适配层设计能力
5. 显式加入“模型适配层”与“Core ML 优化分析”两条交付线

整体实施链路锁定为：

1. `C++` 负责数据适配、清洗、标准化和特征提取
2. `Python / Colab` 负责训练、比较、导出和转换验证
3. `Core ML` 负责 macOS 本地推理
4. `SwiftUI + Charts` 负责展示图表和推理结果

交付原则：

1. 先锁定 schema、工件契约和黄金样本，再进入实现
2. 先打通 Kaggle -> 特征包 -> 训练 -> Core ML -> Swift 的完整闭环
3. 每个 milestone 都必须产出可验收交付件和问题分析记录

## Assumptions & Decisions

### 已锁定决策

1. 当前只规划 `MVP` 阶段，不展开中期和远期实现
2. 计划按“里程碑交付”组织，而不是按周排期组织
3. MVP 只要求单夜离线分析闭环，不要求实时推理
4. MVP 只要求 macOS 端完成展示，不要求 Windows 端 GUI
5. 训练环境默认采用 `Google Colab`
6. `Kaggle` 公开睡眠数据集承担 MVP 主训练数据角色
7. 合成数据承担字段补齐、联调、黄金样本和兜底数据角色
8. `Swift` 端只接收“预计算特征包 + 展示用时序”，不承担原始时序特征提取
9. 主任务锁定为 `recovery_label` 三分类
10. 主部署模型锁定为 `XGBoost / LightGBM`
11. `Logistic Regression` 和 `Random Forest` 作为对比模型保留
12. 每个里程碑都必须附带“问题分析与解决记录”

### 默认实现边界

1. 合成数据仅用于工程和联调验证，不作为效果论证依据
2. 回归任务不进入 MVP 主验收，只作为后续增强方向
3. Core ML 阶段优先关注输入输出对齐、部署可行性和一致性验证
4. Swift 端优先做分析展示页，不优先做复杂交互与历史管理
5. MVP 必须显式体现：
   - 数据适配问题分析
   - 多模型比较与取舍
   - Core ML 转换与优化分析

### 关键实现决策

1. 跨平台工件契约必须在 `MVP-1` 锁定，至少包含：
   - `schema_version`
   - 列顺序
   - CSV 编码
   - 时间格式
   - 标签映射
   - 模型输入特征顺序
   - 数据集来源标记
   - 模型版本
2. 模型适配层在 `MVP-3` 设计，职责包括：
   - 统一不同模型的输入特征顺序
   - 统一标签映射
   - 统一模型元数据
   - 统一导出前中间表示
3. `Swift` 端的“关键影响因素列表”先采用可落地方案：
   - 使用主模型的可提取特征贡献近似
   - 或使用离线预计算解释字段

## Proposed Changes

### 里程碑总览

MVP 阶段保持 4 个里程碑：

1. `MVP-1：数据打底`
2. `MVP-2：预处理跑通`
3. `MVP-3：训练、适配与评估`
4. `MVP-4：端侧部署与优化分析`

各里程碑之间为严格前置依赖关系：

1. `MVP-2` 依赖 `MVP-1`
2. `MVP-3` 依赖 `MVP-2`
3. `MVP-4` 依赖 `MVP-3`

同时增加一条贯穿式交付线：

1. 每个 milestone 必须输出一份“问题分析与解决记录”

### MVP-1：数据打底

#### 目标

完成数据契约、Kaggle 数据集选择、字段映射、黄金样本和跨平台工件契约，为后续模块提供稳定输入。

#### 计划涉及的文件与产物

执行阶段应创建或补充以下内容：

1. `data/raw/`
2. `data/normalized/`
3. `data/features/`
4. `data/labels/`
5. `data/samples/`
6. `docs/` 下的数据字段说明补充文档
7. 样例数据文件：
   - `night_sessions.csv`
   - `heart_rate_timeseries.csv`
   - `activity_timeseries.csv`
8. `artifact manifest` 设计说明
9. `Kaggle` 候选数据集评估记录
10. 黄金样本包说明

#### 具体工作

1. 固定 4 张核心表的字段名、类型、单位和主键关系
2. 明确时间戳格式统一为 `ISO 8601`
3. 明确 `session_id` 作为跨表关联键
4. 锁定样本数据约束：
   - 心率单位为 `bpm`
   - 活动值归一化到 `0-1`
   - 标签值采用固定整数枚举
5. 锁定 MVP 主标签仅使用 `recovery_label`
6. 评估并选择 1 个 `Kaggle` 主数据集，检查：
   - 字段覆盖情况
   - 时间粒度
   - 许可约束
   - 是否存在可用标签或弱标签生成基础
7. 为选定 `Kaggle` 数据集设计字段映射方案
8. 生成至少 `200` 晚合成会话数据的目标规范，作为补齐和兜底来源
9. 为每张核心表准备少量人工可读的样例记录
10. 设计跨平台 `artifact manifest`，明确：
   - `schema_version`
   - 列顺序
   - CSV 编码
   - 时间戳规范
   - 标签枚举
   - 模型输入顺序
   - 数据来源标记
11. 生成黄金样本包定义：
   - 1 晚标准化时序
   - 1 份特征包
   - 1 份 Python 期望预测结果
12. 输出一份“数据集选择与字段映射问题分析记录”

#### 输入

1. 当前方案文档中的 schema 定义
2. 已确认的 MVP 场景与标签规则
3. `Kaggle` 公开睡眠数据选型意图

#### 输出

1. 固定 schema 文档
2. 最小样本 CSV
3. 合成数据生成约束说明
4. `Kaggle` 数据到统一 schema 的映射方案
5. `artifact manifest` 草案
6. 黄金样本包说明
7. 已选定的 `Kaggle` 数据集评估结论
8. 数据问题分析记录

#### 验收标准

1. 后续模块不需要再修改字段命名
2. 四张表之间关联关系明确
3. 可以用样本数据手动追溯一晚会话的完整链路
4. 标签定义不会在训练和展示阶段产生歧义
5. 外部公开数据的字段映射方向已经明确，不会在接入时推翻当前 schema
6. 已经选定 1 个可进入 MVP 的 `Kaggle` 数据集，或明确其不足点与补齐策略
7. 黄金样本包足以支撑后续一致性验证

#### 风险与控制

1. 风险：选到的 `Kaggle` 数据集缺少关键标签或采样粒度不匹配
2. 控制：在本里程碑内完成“适合主训练 / 仅可映射 / 需补齐”的结论，并固化补齐策略

### MVP-2：预处理跑通

#### 目标

完成 `C++` 数据适配、标准化、合成补齐和特征提取 CLI，产出统一特征包。

#### 计划涉及的文件与产物

执行阶段应创建或补充以下内容：

1. `cpp/` 目录及其构建配置
2. `Kaggle` 数据适配模块源码
3. 合成数据生成器源码
4. 预处理 CLI 源码
5. `data/normalized/` 中间结果
6. `data/features/derived_features.csv`
7. 标准化时序样本文件

#### 具体工作

1. 实现 `Kaggle` 主数据集的适配入口：
   - 原始字段读取
   - 时间戳归一化
   - 单位归一化
   - 缺失值和异常值处理
2. 实现合成会话生成逻辑：
   - 睡眠总时长
   - 入睡时间
   - 中断次数
   - 基线心率
3. 实现心率时序生成逻辑：
   - 入睡后下降趋势
   - REM 段波动
   - 觉醒扰动
   - 随机噪声
4. 实现活动时序生成逻辑：
   - 夜间低活动基线
   - 翻身峰值
   - 觉醒片段峰值
5. 实现标准化输出：
   - 原始层到标准化层的转换
   - 缺失值和异常值标记
6. 实现 MVP 特征提取：
   - `mean_hr`
   - `min_hr`
   - `max_hr`
   - `hr_std`
   - `hr_drop_first_90m`
   - `activity_mean`
   - `activity_spike_count`
   - `sleep_fragmentation_score`
   - `total_duration_min`
7. 输出统一特征包：
   - `derived_features.csv`
   - 图表展示用标准化时序文件
8. 输出一份“预处理与数据补齐问题分析记录”

1. `MVP-1` 锁定的 schema
2. 合成规则说明
3. 选定 `Kaggle` 数据集的字段映射方案

#### 输出

1. 可重复执行的 C++ CLI
2. 一批标准化后的时序数据
3. 一份可直接喂给训练脚本的特征表
4. 一份可直接用于 Swift 图表展示的时序样本
5. 预处理问题分析记录

#### 验收标准

1. 同一输入配置能稳定生成同结构输出
2. 输出 CSV 可被 Python 侧直接读取
3. 至少 10 个 MVP 特征被正确写出
4. 预处理链路中不存在字段缺失或跨表关联断裂
5. `Kaggle` 数据和合成数据都能落到同一特征包结构

#### 风险与控制

1. 风险：Kaggle 原始数据与目标 schema 差异过大
2. 控制：将适配逻辑与合成补齐逻辑分层，并在该里程碑结束时形成可审阅的问题清单

### MVP-3：训练、适配与评估

#### 目标

在 `Google Colab` 上完成多模型训练、对比、适配层设计和 Core ML 转换可行性验证。

#### 计划涉及的文件与产物

执行阶段应创建或补充以下内容：

1. `python/` 目录
2. 训练 notebook 或训练脚本
3. 评估结果文件
4. `models/` 下的基线模型产物
5. 模型输入输出说明文档
6. 模型适配层设计文档
7. Core ML 导出实验记录

#### 具体工作

1. 读取 `derived_features.csv`
2. 拆分训练、验证、测试集合
3. 训练 3 类分类模型：
   - `Logistic Regression`
   - `Random Forest`
   - `XGBoost / LightGBM`
4. 比较三类模型在以下维度的差异：
   - 分类效果
   - 特征可解释性
   - Core ML 转换复杂度
   - macOS 部署风险
5. 设计并固定模型适配层，统一：
   - 特征顺序
   - 模型元数据
   - 输出标签映射
   - 导出所需配置
6. 以 `XGBoost / LightGBM` 为主部署候选，完成至少一次 Core ML 转换验证
7. 输出评估结果：
   - 分类准确率
   - 混淆矩阵
   - 关键特征重要性
8. 记录最终采用的输入特征列表和顺序
9. 输出一份“模型选择与适配分析记录”
10. 输出一份“Core ML 转换问题与优化建议记录”，至少覆盖：
   - 输入输出命名
   - 概率输出结构
   - 数值类型与精度
   - 模型大小与可部署性

#### 输入

1. `MVP-2` 输出的 `derived_features.csv`
2. 锁定的标签规则
3. `artifact manifest`

#### 输出

1. 第一版分类模型
2. 训练评估记录

4. 一份模型适配层设计说明
5. 一份 Core ML 转换实验记录
6. 一份模型选择与问题分析记录

#### 验收标准

1. 训练脚本能够从特征表完整跑通
2. 模型输入特征顺序已经固定
3. 评估指标可重复输出

5. 至少完成一次 `XGBoost / LightGBM` 到 Core ML 的可行性验证

#### 风险与控制

1. 风险：`XGBoost / LightGBM` 路线在 Core ML 转换上出现结构或兼容性阻碍
2. 控制：在本里程碑内完成转换试验和风险归因，保证问题被提前暴露，而不是拖到 Swift 集成阶段
