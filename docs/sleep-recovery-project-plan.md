# 睡眠恢复评估项目方案 v1

## 1. 项目目标

本项目围绕“睡眠恢复评估”场景构建一条完整的数据与模型处理链路，目标是在没有硬件设备的前提下，先通过合成数据与公开数据完成 MVP，随后逐步扩展到更真实的数据来源与更强的端侧体验。

项目整体链路如下：

1. 使用 C++ 在 Windows 或 macOS 上完成数据解析、清洗、时间对齐与特征提取。
2. 使用 Python 在 Google Colab 中完成训练、评估与模型导出。
3. 使用 Core ML 在 macOS 端执行本地推理。
4. 使用 SwiftUI + Swift Charts 展示睡眠曲线、恢复等级、疲劳分数与影响因素。

本项目的首要目标不是医疗诊断，而是做出一套可运行、可扩展、可演示的健康数据端侧 AI 原型系统。

## 2. 场景定义

### 2.1 场景名称

个人睡眠恢复分析

### 2.2 核心问题

给定某一晚的多源睡眠相关数据，系统输出：

1. 恢复等级：差 / 一般 / 良好
2. 疲劳分数：0 - 100
3. 关键影响因素：例如夜间活动过多、平均心率偏高、睡眠时长不足等

### 2.3 典型输入

适用于以下一类或多类数据：

1. 夜间心率时序
2. HRV 摘要或 HRV 时序
3. 活动 / 体动时序
4. 睡眠会话摘要
5. 用户主观标签，如起床后疲劳评分

## 3. 数据来源策略

在没有硬件设备的前提下，数据来源分为三层，按优先级从高到低如下。

### 3.1 A 层：合成数据

用途：

1. 打通 C++ 预处理链路
2. 打通 Colab 训练脚本
3. 打通 Core ML 导出与 Swift 展示
4. 验证整体 schema 是否合理

特点：

1. 启动最快
2. 可控性最高
3. 不用于证明模型真实医学效果

### 3.2 B 层：半合成数据

用途：

1. 将公开时序数据适配到统一 schema
2. 根据规则生成弱标签
3. 训练第一版可解释基线模型

特点：

1. 兼顾真实性与工程可控性
2. 非常适合 MVP 之后的增强阶段

### 3.3 C 层：公开真实数据

候选来源方向：

1. PhysioNet
2. Kaggle 睡眠或可穿戴数据集
3. 论文附带开放数据

用途：

1. 验证泛化能力
2. 检查 schema 对真实数据的兼容性
3. 为后续多模态建模做准备

## 4. 数据方案

### 4.1 MVP 数据范围

MVP 阶段只保留最小但完整的三类核心数据：

1. 夜间心率时序
2. 活动强度时序
3. 主观恢复标签

可选增强字段：

1. HRV
2. 睡眠开始和结束时间
3. 夜间中断次数
4. 呼吸频率

### 4.2 统一数据层设计

项目采用“原始层 -> 标准化层 -> 特征层 -> 标签层”的四层结构。

#### 原始层

保存未经处理的输入文件，例如：

1. `csv`
2. `json`
3. 第三方导出的日志文件

#### 标准化层

将不同来源统一成同一套字段命名、时间格式与单位。

#### 特征层

对标准化后的时序数据做统计和窗口特征提取，供训练和推理使用。

#### 标签层

保存分类标签和回归标签。

│  ├─ raw/
│  ├─ normalized/
│  ├─ features/

### 4.4 统一表结构

建议采用以下 4 张核心表。

#### 1. 会话表 `night_sessions.csv`

每一行代表一晚睡眠会话。

字段定义：

| 字段名 | 类型 | 说明 |
|---|---|---|
| session_id | string | 唯一会话 ID |
| user_id | string | 用户 ID，MVP 可固定为 demo_user |
| source_type | string | synthetic / public / manual |
| sleep_start | datetime | 入睡时间 |
| sleep_end | datetime | 起床时间 |
| total_duration_min | float | 总睡眠时长（分钟） |
| awaken_count | int | 夜间中断次数 |
| subjective_fatigue_score | int | 主观疲劳评分，范围建议 1 - 5 |
| recovery_label | int | 恢复等级标签，0 差，1 一般，2 良好 |

#### 2. 心率时序表 `heart_rate_timeseries.csv`

每一行代表一个时间点的心率观测。

字段定义：

| 字段名 | 类型 | 说明 |
|---|---|---|
| session_id | string | 对应会话 ID |
| timestamp | datetime | 时间戳，统一使用 ISO 8601 |
| heart_rate | float | 心率值，单位 bpm |
| quality_flag | int | 质量标记，0 正常，1 缺失插补，2 异常修正 |

#### 3. 活动时序表 `activity_timeseries.csv`

每一行代表一个时间点的活动强度。

字段定义：

| 字段名 | 类型 | 说明 |
|---|---|---|
| session_id | string | 对应会话 ID |
| timestamp | datetime | 时间戳 |
| activity_level | float | 活动强度，范围建议 0 - 1 或原始单位归一化 |
| movement_event | int | 是否发生明显活动峰值，0 否，1 是 |

#### 4. 特征表 `derived_features.csv`

每一行代表一晚会话对应的一组模型输入特征。

字段定义：

| 字段名 | 类型 | 说明 |
|---|---|---|
| session_id | string | 对应会话 ID |
| mean_hr | float | 夜间平均心率 |
| min_hr | float | 夜间最低心率 |
| max_hr | float | 夜间最高心率 |
| hr_std | float | 心率标准差 |
| hr_drop_first_90m | float | 前 90 分钟心率下降幅度 |
| activity_mean | float | 平均活动强度 |
| activity_spike_count | int | 活动峰值次数 |
| sleep_fragmentation_score | float | 睡眠碎片化分数 |
| total_duration_min | float | 睡眠总时长 |
| recovery_label | int | 分类标签 |
| fatigue_score | float | 回归标签，范围建议 0 - 100 |

### 4.5 标签设计

MVP 同时保留一套分类标签和一套回归标签。

#### 分类标签

`recovery_label`

1. `0`：恢复较差
2. `1`：恢复一般
3. `2`：恢复良好

#### 回归标签

`fatigue_score`

范围建议为 `0 - 100`，分数越高表示恢复越好，或越低表示疲劳越重。为避免语义混乱，建议在项目中统一采用“恢复分数”表述，即分数越高越好。

### 4.6 MVP 标签生成规则

在没有真实标签时，可先使用规则生成。

示例规则：

1. 睡眠时长小于 360 分钟，扣分
2. 夜间平均心率偏高，扣分
3. 前 90 分钟心率下降不明显，扣分
4. 夜间活动峰值过多，扣分
5. 夜间中断次数过多，扣分

可参考一条简化评分逻辑：

```text
recovery_score = 100
- duration_penalty
- mean_hr_penalty
- activity_penalty
- awaken_penalty

if recovery_score >= 75 -> recovery_label = 2
if 50 <= recovery_score < 75 -> recovery_label = 1
if recovery_score < 50 -> recovery_label = 0
```

## 5. 合成数据生成方案

### 5.1 合成目标

生成“看起来接近真实夜间睡眠”的多晚数据，用于：

1. 训练和验证基线模型
2. 检查图表展示效果
3. 检查 C++ 管道的稳定性

### 5.2 会话生成规则

每晚会话可随机生成以下内容：

1. 睡眠总时长：330 - 540 分钟
2. 入睡时间：21:30 - 01:30
3. 基线心率：48 - 78 bpm
4. 中断次数：0 - 6
5. 活动峰值：0 - 12 次

### 5.3 心率生成逻辑

按照近似夜间生理趋势生成：

1. 入睡后前 60 - 90 分钟心率逐渐下降
2. 中段相对稳定
3. REM 段波动稍大
4. 觉醒片段出现短时升高

可以按 1 分钟粒度生成：

1. 基础趋势项
2. 周期波动项
3. 随机噪声项
4. 觉醒事件扰动项

### 5.4 活动生成逻辑

活动值以低值为主，在以下时刻出现峰值：

1. 入睡前后短时调整姿势
2. 夜间翻身
3. 中断或醒来

### 5.5 生成规模建议

MVP 阶段建议生成：

1. `200 - 1000` 晚合成会话
2. 每晚按 `1 分钟` 粒度生成时序

这样既足够训练基线模型，也不会让首版工程过重。

## 6. 预处理与特征工程方案

### 6.1 C++ 预处理职责

C++ 模块负责：

1. 读取原始数据或合成数据
2. 校验字段完整性
3. 统一时间戳格式
4. 重采样到固定粒度
5. 异常值过滤
6. 缺失值标记与有限插补
7. 特征提取
8. 输出标准化 CSV

### 6.2 MVP 必做特征

全局统计特征：

1. 平均心率
2. 最低心率
3. 最高心率
4. 心率标准差
5. 平均活动值
6. 活动峰值次数
7. 睡眠总时长
8. 夜间中断次数

分段特征：

1. 前 90 分钟平均心率
2. 后半夜平均心率
3. 心率下降幅度
4. 高活动片段数量

### 6.3 中期扩展特征

1. HRV 统计特征
2. 个体基线偏差特征
3. 周期性与波动性特征
4. 多夜滚动窗口特征

## 7. 模型方案

### 7.1 MVP 模型

优先选择轻量、可解释、易导出的模型：

1. Logistic Regression
2. Random Forest
3. XGBoost 或 LightGBM

建议先做两类模型：

1. 分类模型：预测 `recovery_label`
2. 回归模型：预测 `recovery_score`

### 7.2 中期模型

在时序质量提升后，再尝试：

1. 1D CNN
2. LSTM
3. Temporal Convolution

### 7.3 模型导出

推荐流程：

1. 在 Colab 完成训练
2. 导出为通用格式
3. 使用 `coremltools` 转换为 `.mlpackage`
4. 在 macOS 上验证输入输出与 Swift 端一致

## 10. 阶段计划

### 10.1 MVP 阶段

目标：在没有真实硬件的前提下，完成一条可运行的端到端闭环。

#### MVP 范围

1. 数据来源以合成数据为主
2. 输入只保留心率、活动、主观标签
3. 训练基线模型
4. 导出 Core ML 模型
5. 在 macOS 上完成单夜离线分析与图表展示

#### MVP 交付物

1. `C++` 合成数据生成器
2. `C++` 预处理与特征提取 CLI
3. `Colab` 训练 notebook 或脚本
4. `Core ML` 导出模型
5. `SwiftUI` macOS 演示应用
6. 文档：字段说明、模型输入输出说明、运行说明

#### MVP 推荐里程碑

`MVP-1：数据打底`

1. 固定统一 schema
2. 生成 200 晚以上的合成数据
3. 输出样例 CSV

`MVP-2：预处理跑通`

1. 完成 C++ CLI
2. 完成时间对齐与特征提取
3. 输出 `derived_features.csv`

`MVP-3：训练与评估`

1. 在 Colab 训练分类或回归模型
2. 输出混淆矩阵或误差指标
3. 固定第一版模型输入特征

`MVP-4：端侧部署`

1. 转换为 Core ML
2. 在 macOS 应用中完成推理
3. 展示心率曲线与恢复结果

#### MVP 成功标准

1. 能从一份原始或合成夜间数据生成一份预测结果
2. Swift 端可以离线展示完整分析结果
3. 训练端与推理端使用同一套特征定义
4. 所有输入输出文件都有固定 schema

### 10.2 中期阶段

目标：增强真实性、提升模型质量、扩展到更多数据模态。

#### 中期重点

1. 接入公开真实数据集
2. 引入 HRV、睡眠摘要等字段
3. 增加多夜趋势分析
4. 增加解释性能力
5. 提升模型稳定性和泛化表现

#### 中期交付物

1. 公开数据适配层
2. 更完整的特征工程模块
3. 7 天或 14 天趋势分析页面
4. 多模型评估报告

### 10.3 远期阶段

目标：从演示型系统走向具备更强产品化能力的平台。

#### 远期重点

1. 支持多源数据导入
2. 支持用户个体基线建模
3. 支持更细粒度的睡眠片段分析
4. 支持模型更新与版本管理
5. 支持更实时或近实时的端侧分析

#### 远期方向

1. 接入真实设备数据
2. 探索时序深度学习模型
3. 构建个性化恢复评分系统
4. 构建更完整的本地数据管理与历史对比能力

## 11. 当前优先级建议

当前最优先做的不是追求复杂模型，而是保证 MVP 能稳定闭环。

建议按以下顺序推进：

1. 固定 schema 与字段命名
2. 先写合成数据生成器
3. 再写 C++ 预处理与特征提取
4. 在 Colab 建立基线训练脚本
5. 最后接入 macOS Swift 展示

## 12. MVP 下一步执行建议

如果要马上开始实现，建议第一周只做以下内容：

1. 创建项目目录结构
2. 输出一批合成样例数据
3. 固定 10 - 15 个 MVP 特征
4. 在 Colab 跑出第一个分类模型
5. 明确 Core ML 输入输出字段

完成上述内容后，就已经具备进入应用展示层开发的基础。
